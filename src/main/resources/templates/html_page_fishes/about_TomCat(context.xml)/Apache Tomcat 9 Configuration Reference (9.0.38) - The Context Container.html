Знакомство

В описании ниже используется переменное имя $CATALINA-BASE для обозначения базового каталога, на котором разрешено большинство относительных путей. Если вы не настроили Tomcat для нескольких экземпляров, установив каталог CATALINA_BASE, то $CATALINA-BASE будет установлен на значение $CATALINA-HOME, каталог, в который вы установили Tomcat.

Элемент Context представляет собой веб-приложение,которое работает в рамках конкретного виртуального узла. Каждое веб-приложение основано на файле Web Application Archive (WAR) или соответствующем каталоге, содержащем соответствующее распакованое содержимое, как описано в спецификации Servlet (версия 2.2 или позже). Для получения дополнительной информации об архивах веб-приложений, вы можете скачать спецификации Servlet, и обзор Tomcat Приложение Разработчик Руководство.

Веб-приложение, используемое для обработки каждого запроса HTTP, выбирается Каталиной на основе сопоставления как можно более длинной префикса запроса URI с контекстным путем каждого определенного контекста. После выбора этот контекст выберет соответствующий сервлет для обработки входящего запроса в соответствии с отображением серверов, определяемых развертыванием веб-приложений.

Вы можете определить столько элементов контекста, сколько пожелаете. Каждый такой контекст должен иметь уникальное контексте имя в виртуальном хосте. Контекстный путь не должен быть уникальным (см. параллельное развертывание ниже). Кроме того, контекст должен присутствовать с контекстным путем, равным строке нулевой длины. Этот контекст становится веб-приложением по умолчанию для этого виртуального узла и используется для обработки всех запросов, которые не соответствуют никакому другому контексту context path.

Параллельное развертывание

Можно развертывать несколько версий веб-приложения с одним и тем же контекстом одновременно. Правила, используемые для сопоставить запросы с контекстной версией, следующие:

  • Если в запросе нет информации о сеансе, используйте последнюю версию.
  • Если в запросе присутствует информация о сеансе, проверьте сеанс-менеджера каждой версии для соответствующего сеанса, и если она найдена, используйте эту версию.
  • Если в запросе присутствует информация о сеансе, но не найдено соответствующей сессии, используйте последнюю версию.

Хост может быть настроен (через ) для удаления старых версий, развернутых таким образом, как только они больше не используются.undeployOldVersions

Определение контекста

Не рекомендуется помесять элементы напрямую в файл server.xml. Это связано с тем, что это делает изменение конфигурации Context более инвазивным, так как основной файл не может быть перезагружен без перезапуска Tomcat. Элементы контекста по умолчанию (см. ниже) также перезаписать конфигурацию любых элементов, размещенных непосредственно в server.xml. Чтобы предотвратить это, следует установить атрибут элемента,определяемого в server.xml.conf/server.xmloverridetrue

Отдельные элементы контекста могут быть четко определены:

  • В отдельном файле внутри файла приложения. По желанию (на основе атрибута copyXML) хост может быть скопирован и переименован в базовое имя файла приложения плюс расширение ".xml"./META-INF/context.xml$CATALINA_BASE/conf/[enginename]/[hostname]/
  • В отдельных файлах (с расширением ".xml") в каталоге. Контекстный путь и версия будут получены из базового имени файла (имя файла меньше расширения .xml). Этот файл всегда будет иметь приоритет над любым context.xml файл упакован в каталоге META-INF веб-приложения.$CATALINA_BASE/conf/[enginename]/[hostname]/
  • Внутри элемента Хоста в главном.conf/server.xml

По умолчанию могут быть определены элементы контекста, которые применяются к нескольким веб-приложениям. Конфигурация для отдельного веб-приложения переопределит все, что настроено в одном из этих по умолчанию. Любые вложенные элементы, например, элементы lt;Resource,gt; которые определяются в контексте по умолчанию, будут созданы один раз для каждого контекста, к которому применяется по умолчанию. Они не будут разделены между элементами контекста.

  • В файле: информация об элементах Context будет загружена всеми веб-приложениями.$CATALINA_BASE/conf/context.xml
  • В файле: информация об элементе Контекста будет загружена всеми веб-приложениями этого узла.$CATALINA_BASE/conf/[enginename]/[hostname]/context.xml.default

За исключением server.xml, файлы, определяющие элементы контекста, могут определить только один элемент Контекста.

В дополнение к явно указанным элементам Context существует несколько методов, с помощью которых элементы Context могут быть созданы автоматически для вас. Для получения дополнительной информации можно получить подробную информацию об автоматическом развертывании приложений и веб-приложениях пользователей.

Для определения нескольких контекстов, в котором используется один файл или каталог WAR, используйте один из вариантов, описанных в разделе Именования выше, для создания контекста, который имеет путь, не связанный с именем базового файла.

Nested Components

You can nest at most one instance of the following utility components by nesting a corresponding element inside your Context element:

  • Cookie Processor - Configure parsing and generation of HTTP cookie headers.
  • Loader - Configure the web application class loader that will be used to load servlet and bean classes for this web application. Normally, the default configuration of the class loader will be sufficient.
  • Manager - Configure the session manager that will be used to create, destroy, and persist HTTP sessions for this web application. Normally, the default configuration of the session manager will be sufficient.
  • Realm - Configure a realm that will allow its database of users, and their associated roles, to be utilized solely for this particular web application. If not specified, this web application will utilize the Realm associated with the owning Host or Engine.
  • Resources - Configure the resource manager that will be used to access the static resources associated with this web application. Normally, the default configuration of the resource manager will be sufficient.
  • WatchedResource - The auto deployer will monitor the specified static resource of the web application for updates, and will reload the web application if it is updated. The content of this element must be a string.
  • JarScanner - Configure the Jar Scanner that will be used to scan the web application for JAR files and directories of class files. It is typically used during web application start to identify configuration files such as TLDs o web-fragment.xml files that must be processed as part of the web application initialisation. Normally, the default Jar Scanner configuration will be sufficient.

Logging

A context is associated with the log category. Note that the brackets are actually part of the name, don't omit them.org.apache.catalina.core.ContainerBase.[enginename].[hostname].[path]

Access Logs

When you run a web server, one of the output files normally generated is an access log, which generates one line of information for each request processed by the server, in a standard format. Catalina includes an optional Valve implementation that can create access logs in the same standard format created by web servers, or in any number of custom formats.

You can ask Catalina to create an access log for all requests processed by an Engine, Host, or Context by nesting a Valve element like this:

<Context>
  ...
  <Valve className="org.apache.catalina.valves.AccessLogValve"
         prefix="localhost_access_log" suffix=".txt"
         pattern="common"/>
  ...
</Context>

See Access Logging Valves for more information on the configuration attributes that are supported.

Automatic Context Configuration

If you use the standard Context implementation, the following configuration steps occur automatically when Catalina is started, or whenever this web application is reloaded. No special configuration is required to enable this feature.

  • If you have not declared your own Loader element, a standard web application class loader will be configured.
  • If you have not declared your own Manager element, a standard session manager will be configured.
  • If you have not declared your own Resources element, a standard resources manager will be configured.
  • The web application properties listed in will be processed as defaults for this web application. This is used to establish default mappings (such as mapping the extension to the corresponding JSP servlet), and other standard features that apply to all web applications.conf/web.xml*.jsp
  • The web application properties listed in the resource for this web application will be processed (if this resource exists), taking precedence over the defaults./WEB-INF/tomcat-web.xml
  • The web application properties listed in the resource for this web application will be processed (if this resource exists)./WEB-INF/web.xml
  • If your web application has specified security constraints that might require user authentication, an appropriate Authenticator that implements the login method you have selected will be configured.

Request Filters

You can ask Catalina to check the IP address, or host name, on every incoming request directed to the surrounding Engine, Host, or Context element. The remote address or name will be checked against configured "accept" and/or "deny" filters, which are defined using Regular Expression syntax. Requests that come from locations that are not accepted will be rejected with an HTTP "Forbidden" error. Example filter declarations:java.util.regex

<Context>
  ...
  <Valve className="org.apache.catalina.valves.RemoteHostValve"
         allow=".*\.mycompany\.com|www\.yourcompany\.com"/>
  <Valve className="org.apache.catalina.valves.RemoteAddrValve"
         deny="192\.168\.1\.\d+"/>
  ...
</Context>

See Remote Address Filter and Remote Host Filter for more information about the configuration options that are supported.
